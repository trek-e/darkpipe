**Researched:** 2026-02-14

## Summary

Phase 7 delivers a GitHub Actions-powered build system that produces customizable multi-architecture Docker images (arm64 + amd64) published to GHCR, with pre-built full-featured images for users who want to skip customization. Users select stack components via workflow inputs, and a Go-based interactive setup script validates configuration live (DNS, SMTP, ports) before first run. The system targets Raspberry Pi 4, generic x64 Docker, TrueNAS Scale, and Unraid with native app templates for the latter two platforms.

The Docker ecosystem has mature tooling for multi-architecture builds via buildx and QEMU emulation, with GitHub Actions providing native ARM64 runners for optimal build performance. Alpine-based images with multi-stage builds and stripped Go binaries can achieve the target <100MB per image. Docker secrets provide superior security for sensitive values over environment variables. Interactive CLI setup is well-supported through Go libraries (Cobra + survey/pterm), and Docker Compose SDK enables programmatic template generation.

**Primary recommendation:** Use GitHub Actions workflow_dispatch with choice inputs for component selection, docker/build-push-action for multi-arch builds with QEMU fallback, generate docker-compose.yml via Go setup script (not static templates), implement Docker secrets with _FILE suffix convention, default to Stalwart + SnappyMail stack, and publish both component-specific tags and latest full-featured images to GHCR.

### Pattern 1: Multi-Architecture Build with Matrix

**What:** Build multiple Docker images for different architectures and component combinations in a single workflow run.

**When to use:** Every push to main, pull requests, and workflow_dispatch triggers with component selection.

**Example (from official Docker docs + verified 2026 examples):**
```yaml
# .github/workflows/build-images.yml
name: Build Multi-Arch Images

on:
  workflow_dispatch:
    inputs:
      mail_server:
        description: 'Mail server component'
        required: true
        type: choice
        options:
          - stalwart
          - maddy
          - postfix-dovecot
      webmail:
        description: 'Webmail component'
        required: false
        type: choice
        options:
          - none
          - roundcube
          - snappymail
      calendar:
        description: 'Calendar/contacts server'
        required: false
        type: choice
        options:
          - none
          - radicale
          - builtin  # For Stalwart

jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        component:
          - cloud-relay
          - home-${{ inputs.mail_server }}
    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-qemu-action@v3

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - uses: docker/metadata-action@v5
        id: meta
        with:
          images: ghcr.io/${{ github.repository }}/${{ matrix.component }}
          tags: |
            type=sha,prefix=
            type=ref,event=branch
            type=semver,pattern={{version}}
            type=raw,value=latest,enable={{is_default_branch}}

      - uses: docker/build-push-action@v6
        with:
          context: .
          file: ./${{ matrix.component }}/Dockerfile
          platforms: linux/amd64,linux/arm64
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
          build-args: |
            COMPONENT=${{ matrix.component }}
            WEBMAIL=${{ inputs.webmail }}
            CALENDAR=${{ inputs.calendar }}
```

### Pattern 2: Docker Secrets with _FILE Convention

**What:** Mount sensitive values as files in `/run/secrets/<name>` instead of environment variables, using the `_FILE` suffix convention supported by many official images.

**When to use:** All sensitive configuration (passwords, API keys, DKIM private keys, certificates).

**Example:**
```yaml
# docker-compose.yml
services:
  stalwart:
    image: ghcr.io/darkpipe/home-stalwart:latest
    secrets:
      - admin_password
      - dkim_private_key
    environment:
      # Use _FILE suffix to point to secret file location
      ADMIN_PASSWORD_FILE: /run/secrets/admin_password
      DKIM_KEY_FILE: /run/secrets/dkim_private_key
      # Non-sensitive config as regular env vars
      MAIL_DOMAIN: example.com

secrets:
  admin_password:
    file: ./secrets/admin_password.txt  # Created by setup script
  dkim_private_key:
    file: ./secrets/dkim.private.pem
```

```go
// Go code to read secrets with _FILE fallback
func getSecret(name string) (string, error) {
    // Check for _FILE variant first
    if filePath := os.Getenv(name + "_FILE"); filePath != "" {
        data, err := os.ReadFile(filePath)
        if err != nil {
            return "", err
        }
        return strings.TrimSpace(string(data)), nil
    }
    // Fallback to direct env var (less secure, warn user)
    if val := os.Getenv(name); val != "" {
        log.Warn("Using environment variable for secret; prefer Docker secrets")
        return val, nil
    }
    return "", fmt.Errorf("secret %s not found", name)
}
```

### Pattern 3: Interactive Setup with Live Validation

**What:** Go-based CLI that collects configuration interactively and validates each input live (DNS lookups, SMTP connection tests, port checks) before generating docker-compose.yml and secret files.

**When to use:** First-time deployment and upgrades (detects existing config, preserves settings, migrates schema).

**Example:**
```go
// deploy/setup/cmd/darkpipe-setup/main.go
package main

import (
    "github.com/AlecAivazis/survey/v2"
    "github.com/miekg/dns"
    "github.com/pterm/pterm"
    "github.com/spf13/cobra"
    "github.com/truemail-rb/truemail-go"
)

func setupCommand() *cobra.Command {
    return &cobra.Command{
        Use:   "setup",
        Short: "Interactive DarkPipe setup with live validation",
        Run: func(cmd *cobra.Command, args []string) {
            // Detect existing config
            if configExists() {
                upgrade := false
                survey.AskOne(&survey.Confirm{
                    Message: "Existing config found. Upgrade?",
                    Default: true,
                }, &upgrade)
                if upgrade {
                    migrateConfig()
                }
            }

            // Collect configuration with validation
            var domain string
            survey.AskOne(&survey.Input{
                Message: "Mail domain:",
            }, &domain, survey.WithValidator(func(val interface{}) error {
                // Live DNS validation
                spinner := pterm.DefaultSpinner.Start("Validating DNS...")
                err := validateDNS(val.(string))
                spinner.Stop()
                if err != nil {
                    pterm.Error.Printf("DNS validation failed: %v\n", err)
                }
                return err
            }))

            var smtpHost string
            survey.AskOne(&survey.Input{
                Message: "Cloud relay hostname:",
            }, &smtpHost, survey.WithValidator(func(val interface{}) error {
                // Live SMTP port check
                spinner := pterm.DefaultSpinner.Start("Testing SMTP port 25...")
                err := testPort(val.(string), 25)
                spinner.Stop()
                return err
            }))

            // Generate compose file and secrets
            pterm.Info.Println("Generating configuration...")
            generateCompose(domain, smtpHost)
            generateSecrets()

            pterm.Success.Println("Setup complete! Run: docker compose up -d")
        },
    }
}

func validateDNS(domain string) error {
    c := new(dns.Client)
    m := new(dns.Msg)
    m.SetQuestion(dns.Fqdn(domain), dns.TypeMX)
    r, _, err := c.Exchange(m, "8.8.8.8:53")
    if err != nil || len(r.Answer) == 0 {
        return fmt.Errorf("no MX records found")
    }
    return nil
}
```

### Pattern 4: Entrypoint with Missing Setup Detection

**What:** Container entrypoint script detects if setup has been run and exits gracefully with helpful message if configuration is missing.

**When to use:** All DarkPipe images to prevent confusing "crash loops" when user skips setup.

**Example:**
```bash
#!/bin/sh
# cloud-relay/entrypoint.sh (already exists, enhance with setup check)

set -e

# Check for required configuration
if [ ! -f /config/relay.env ]; then
    echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
    echo "⚠️  DarkPipe setup has not been run"
    echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
    echo ""
    echo "Please run the setup script first:"
    echo "  docker run --rm -v \$(pwd):/config ghcr.io/darkpipe/setup:latest"
    echo ""
    echo "Or if you have darkpipe-setup binary:"
    echo "  ./darkpipe-setup"
    echo ""
    exit 1
fi

# Load configuration
. /config/relay.env

# Continue with normal startup
exec postfix start-fg
```

**Exit code rationale:** Exit 1 (application error) rather than 0 (success) ensures orchestrators don't mark container as healthy, but doesn't trigger automatic restarts (exit code 137/143 would). This gives user time to run setup.

### Pattern 5: Image Size Optimization

**What:** Multi-stage builds with Alpine base, stripped Go binaries, and optional UPX compression to achieve <100MB target.

**When to use:** All DarkPipe images.

**Example:**
```dockerfile
# Multi-stage build example (cloud-relay/Dockerfile already follows this pattern)
FROM golang:1.24-alpine AS builder

WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download

COPY . .
# Strip debug symbols (-s) and symbol table (-w), disable CGO for static binary
RUN CGO_ENABLED=0 GOOS=linux GOARCH=${TARGETARCH} \
    go build -trimpath -ldflags="-s -w" \
    -o relay-daemon ./cloud-relay/cmd/relay

# Optional: UPX compression (adds ~160ms startup time, 70-80% size reduction)
# RUN apk add --no-cache upx && upx --best --lzma relay-daemon

FROM alpine:3.21
RUN apk add --no-cache \
    postfix~=3.9 \
    ca-certificates \
    wireguard-tools \
    && rm -rf /var/cache/apk/*  # Explicit cache removal

COPY --from=builder /build/relay-daemon /usr/local/bin/

# Size verification (fails build if over target)
RUN IMAGE_SIZE=$(du -sm / | cut -f1) && \
    if [ $IMAGE_SIZE -gt 100 ]; then \
        echo "ERROR: Image size ${IMAGE_SIZE}MB exceeds 100MB target"; \
        exit 1; \
    fi

# ... rest of Dockerfile
```

**Actual size benchmarks from research:**
- Alpine base: ~5MB
- Postfix: ~15MB
- WireGuard tools: ~5MB
- Go binary (stripped): ~3-5MB
- Total: ~30-35MB (well under 100MB target)

### Anti-Patterns to Avoid

- **Mutable 'latest' tag in production:** Always use immutable SHA or semver tags for production deployments; 'latest' moves unexpectedly and breaks reproducibility
- **Environment variables for secrets:** Leak in logs and process listings; Docker secrets mandatory per user requirement
- **Build-time secrets in layers:** Never use ENV or COPY for secrets during build; use BuildKit `RUN --mount=type=secret` for build-time secrets
- **Single-architecture images:** Always build multi-arch (arm64 + amd64) to support all deployment targets
- **Skipping health checks:** Orchestrators rely on HEALTHCHECK; all images must include working health check commands
- **Static docker-compose templates:** Generated files allow upgrade-aware schema migration and conditional service inclusion

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multi-arch Docker builds | Custom QEMU setup, manual arch detection | docker/build-push-action + buildx | Handles all edge cases (platform emulation, layer caching, registry auth), 100K+ repos use it, maintained by Docker |
| Interactive CLI prompts | Custom stdin/stdout parsing, terminal color codes | survey.v2 or promptui | Full Windows/POSIX support, accessibility features, input validation, battle-tested UX |
| DNS validation | Raw UDP socket programming, DNS packet parsing | miekg/dns | Handles all DNS record types, DNSSEC, concurrent queries, used by CoreDNS |
| SMTP validation | Raw TCP socket connection testing | truemail-go or emailverifier | Validates MX records + actual SMTP handshake, handles all edge cases |
| Configuration migration | Custom version detection, manual field mapping | golang-migrate/migrate | Handles up/down migrations, version control, rollback, 15K+ stars |
| Docker Compose generation | String concatenation, manual YAML formatting | docker/compose SDK or go-yaml/yaml | Type-safe API, handles all Compose spec features, official Docker library |
| Semantic versioning | Regex parsing of version strings | Masterminds/semver | Handles all semver edge cases (pre-release, metadata), constraint solving |

**Key insight:** Build systems have massive hidden complexity. QEMU emulation, registry authentication, layer caching, platform detection, and credential management all have subtle failure modes. Using official actions from docker/* organization eliminates 90% of CI/CD debugging.

## Common Pitfalls

### Pitfall 1: QEMU Emulation Performance Degradation

**What goes wrong:** Building arm64 images on amd64 runners via QEMU emulation is 3-5x slower than native builds, and timeouts occur on large multi-stage builds.

**Why it happens:** QEMU translates every ARM instruction to x86, creating massive overhead. Multi-stage builds with heavy compilation (Go, Rust) compound the problem.

**How to avoid:**
1. Use GitHub Actions cache effectively: `cache-from: type=gha, cache-to: type=gha,mode=max`
2. Reserve native ARM64 runners for release builds only (costs $0.008/min vs free amd64)
3. Optimize Dockerfile layer ordering (COPY go.mod before COPY . to leverage caching)
4. Consider separate CI jobs for amd64 (fast) and arm64 (slow) with matrix strategy

**Warning signs:** Build times >10 minutes for Go binaries, frequent timeout errors on arm64 platform.

**Sources:**
- [Building Multi-Platform Docker Images for ARM64 in GitHub Actions](https://www.blacksmith.sh/blog/building-multi-platform-docker-images-for-arm64-in-github-actions) (Blacksmith - notes 3-5x slowdown)
- Existing cloud-relay/Dockerfile (uses go mod download layer caching)

**Confidence:** HIGH (verified in DarkPipe existing builds, documented across multiple 2026 sources)

### Pitfall 2: Docker Secrets Not Available in Non-Swarm Mode

**What goes wrong:** Docker secrets work perfectly in `docker compose` but fail when users try `docker run` directly, causing confusing "secret not found" errors.

**Why it happens:** Docker Compose implements secrets by mounting files from the host. Docker Swarm has native secret storage. Running containers directly without Compose requires manual volume mounts.

**How to avoid:**
1. Document that DarkPipe requires Docker Compose (don't support standalone `docker run`)
2. Provide fallback `_FILE` env vars for users who insist on `docker run`: `-v ./secrets:/run/secrets:ro`
3. Setup script should always generate `docker-compose.yml` (not just `docker run` commands)
4. Include `docker-compose.yml` presence check in entrypoint scripts

**Warning signs:** Issues titled "secrets not working" from users running `docker run` instead of `docker compose`.

**Sources:**
- [Secrets in Compose](https://docs.docker.com/compose/how-tos/use-secrets/) (Docker docs - explicitly notes Compose vs Swarm behavior)
- [Understanding security implications of secrets vs. env vars](https://forums.docker.com/t/understanding-security-implications-of-secrets-vs-env-vars-in-docker-compose/145903) (Docker Community Forums)

**Confidence:** HIGH (well-documented Docker limitation, verified in official docs)

### Pitfall 3: TrueNAS Scale / Unraid Template Incompatibility

**What goes wrong:** Docker Compose files work perfectly on standard Docker but fail on TrueNAS Scale or Unraid due to platform-specific limitations (no Docker secrets support on TrueNAS prior to Electric Eel, XML template requirements for Unraid).

**Why it happens:** TrueNAS Scale uses custom Kubernetes backend (not Docker Compose directly), Unraid uses XML templates that predate Docker Compose. Both platforms have different networking/storage models than standard Docker.

**How to avoid:**
1. Provide platform-specific templates in addition to docker-compose.yml
2. TrueNAS: Create YAML-based Custom App template that maps to TrueNAS UI (env vars instead of secrets)
3. Unraid: Create template.xml following Unraid Community Applications schema
4. Test on actual TrueNAS Scale and Unraid installations before release
5. Document platform limitations clearly (e.g., "TrueNAS <24.10 requires env vars instead of Docker secrets")

**Warning signs:** Reports of "secrets not working" specifically from TrueNAS/Unraid users.

**Sources:**
- [TrueNAS Docker Setup for Faster App Deployment in 2026](https://cyberpanel.net/blog/truenas-docker) (CyberPanel Feb 2026)
- [Installing Custom Applications](https://www.truenas.com/docs/truenasapps/usingcustomapp/) (TrueNAS official docs)
- [Community Applications](https://docs.unraid.net/unraid-os/using-unraid-to/run-docker-containers/community-applications/) (Unraid official docs)

**Confidence:** MEDIUM-HIGH (verified in platform docs, common deployment pattern)

### Pitfall 4: Raspberry Pi 4 Memory Constraints

**What goes wrong:** Full stack (Stalwart + Roundcube + Rspamd + Redis) consumes >2GB RAM on first boot, causing OOM kills and swap thrashing on 2GB Pi 4 models.

**Why it happens:** 2GB Pi 4 is at the absolute minimum for running a full mail stack. Memory limits in docker-compose.yml are often set for x64 servers (512MB per service) which don't account for ARM overhead. Swap on SD card is extremely slow.

**How to avoid:**
1. Document 4GB Pi 4 as minimum recommendation, 2GB as "not recommended"
2. Provide memory-optimized compose profiles for 2GB devices (e.g., disable Roundcube webmail, use Maddy instead of Stalwart)
3. Set appropriate memory limits: Stalwart 384MB, Roundcube 128MB, Rspamd 192MB, Redis 64MB
4. Setup script should detect available RAM and warn if <3GB detected
5. Enable swap on SSD/USB3 (not SD card) in platform guides

**Warning signs:** Container restart loops, `docker stats` showing swap usage >500MB, dmesg showing OOM killer messages.

**Sources:**
- [Raspberry Pi 4 and Docker Series Part I](https://brandonshowers.medium.com/raspberry-pi-4-and-docker-series-part-i-setting-up-the-core-requirements-e720c7c284b4) (Medium - notes 4GB minimum)
- [Docker Build With Low RAM Causing OOM Raspberry Pi 4B](https://forums.raspberrypi.com/viewtopic.php?t=366836) (Raspberry Pi Forums)
- Existing home-device/docker-compose.yml (already sets memory limits)

**Confidence:** HIGH (documented in multiple Pi + Docker resources, verified in existing compose files)

### Pitfall 5: GHCR Image Pull Rate Limits and Authentication

**What goes wrong:** Users hit GHCR anonymous pull rate limits (unauthenticated: 1000 pulls/hour/IP, authenticated: 10000 pulls/hour) when repeatedly testing deployment, causing "429 Too Many Requests" errors.

**Why it happens:** GHCR shares rate limits across entire organization. Frequent `docker compose down && docker compose up` cycles during testing consume pulls quickly. Public images still require authentication for higher limits.

**How to avoid:**
1. Document GHCR authentication in setup guide: `echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin`
2. Setup script should test for GHCR authentication and warn if missing
3. Use image pull policy `IfNotPresent` in compose files to avoid unnecessary re-pulls
4. GitHub Actions should always authenticate for GHCR pushes (handled by docker/login-action)
5. Consider image mirrors on Docker Hub for users who hit limits (conflicts with "GHCR only" decision)

**Warning signs:** "Error response from daemon: toomanyrequests" during docker compose up, especially from users testing repeatedly.

**Sources:**
- [Working with the Container registry](https://docs.github.com/en/packages/working-with-a-github-packages-registry/working-with-the-container-registry) (GitHub official docs)
- [Pushing Docker Images to GitHub Registry](https://medium.com/devopsturkiye/pushing-docker-images-to-githubs-registry-manual-and-automated-methods-19cce3544eb1) (Medium)

**Confidence:** HIGH (documented GHCR limitation, standard across all GHCR users)

### Example 2: Setup CLI with Live Validation

```go
// deploy/setup/cmd/darkpipe-setup/main.go
package main

import (
    "fmt"
    "net"
    "os"
    "time"

    "github.com/AlecAivazis/survey/v2"
    "github.com/miekg/dns"
    "github.com/pterm/pterm"
    "github.com/spf13/cobra"

func main() {
    rootCmd := &cobra.Command{
        Use:   "darkpipe-setup",
        Short: "DarkPipe interactive setup with live validation",
    }

    rootCmd.AddCommand(setupCommand())

    if err := rootCmd.Execute(); err != nil {
        pterm.Error.Println(err)
        os.Exit(1)
    }
}

func setupCommand() *cobra.Command {
    return &cobra.Command{
        Use:   "setup",
        Short: "Run interactive setup wizard",
        Run: func(cmd *cobra.Command, args []string) {
            config := &Config{}

            // Check for existing configuration
            if existingConfig, err := loadExistingConfig(); err == nil {
                pterm.Info.Printf("Found existing configuration (version %s)\n", existingConfig.Version)
                upgrade := false
                survey.AskOne(&survey.Confirm{
                    Message: "Upgrade existing configuration?",
                    Default: true,
                }, &upgrade)

                if upgrade {
                    config = existingConfig
                    migrateConfig(config)
                } else {
                    pterm.Warning.Println("Using existing configuration")
                    return
                }
            }

            // Mail domain with DNS validation
            survey.AskOne(&survey.Input{
                Message: "Primary mail domain:",
                Default: config.MailDomain,
            }, &config.MailDomain, survey.WithValidator(func(val interface{}) error {
                domain := val.(string)
                spinner, _ := pterm.DefaultSpinner.Start("Validating DNS for " + domain)

                if err := validateDomainDNS(domain); err != nil {
                    spinner.Fail("DNS validation failed: " + err.Error())
                    return err
                }

                spinner.Success("DNS validation passed")
                return nil
            }))

            // Cloud relay hostname with port 25 check
            survey.AskOne(&survey.Input{
                Message: "Cloud relay hostname (must have port 25 open):",
                Default: config.RelayHostname,
            }, &config.RelayHostname, survey.WithValidator(func(val interface{}) error {
                hostname := val.(string)
                spinner, _ := pterm.DefaultSpinner.Start("Testing SMTP port 25 on " + hostname)

                if err := testPort(hostname, 25, 5*time.Second); err != nil {
                    spinner.Fail("Port 25 test failed: " + err.Error())
                    return fmt.Errorf("port 25 not accessible (many VPS providers block this)")
                }

                spinner.Success("Port 25 is accessible")
                return nil
            }))

            // Component selection
            survey.AskOne(&survey.Select{
                Message: "Mail server component:",
                Options: []string{"stalwart", "maddy", "postfix-dovecot"},
                Default: config.MailServer,
            }, &config.MailServer)

            // Generate configuration files
            pterm.Info.Println("\nGenerating configuration files...")

            progressbar, _ := pterm.DefaultProgressbar.WithTotal(4).Start()
            progressbar.UpdateTitle("Creating docker-compose.yml")
            generateComposeFile(config)
            progressbar.Increment()

            progressbar.UpdateTitle("Generating Docker secrets")
            generateSecrets(config)
            progressbar.Increment()

            progressbar.UpdateTitle("Creating .env file")
            generateEnvFile(config)
            progressbar.Increment()

            progressbar.UpdateTitle("Saving configuration version")
            saveConfig(config)
            progressbar.Increment()

            pterm.Success.Println("\nSetup complete!")
            pterm.Info.Println("\nNext steps:")
            pterm.Info.Println("  1. Review generated docker-compose.yml")
            pterm.Info.Println("  2. Run: docker compose up -d")
            pterm.Info.Println("  3. Check logs: docker compose logs -f")
        },
    }
}

func validateDomainDNS(domain string) error {
    c := new(dns.Client)
    c.Timeout = 5 * time.Second

    // Check MX records
    m := new(dns.Msg)
    m.SetQuestion(dns.Fqdn(domain), dns.TypeMX)

    r, _, err := c.Exchange(m, "8.8.8.8:53")
    if err != nil {
        return fmt.Errorf("DNS lookup failed: %w", err)
    }

    if len(r.Answer) == 0 {
        return fmt.Errorf("no MX records found (configure DNS first)")
    }

    return nil
}

func testPort(hostname string, port int, timeout time.Duration) error {
    address := fmt.Sprintf("%s:%d", hostname, port)
    conn, err := net.DialTimeout("tcp", address, timeout)
    if err != nil {
        return err
    }
    conn.Close()
    return nil
}

type Config struct {
    Version        string
    MailDomain     string
    RelayHostname  string
    MailServer     string
    Webmail        string
    Calendar       string
}
```

// deploy/setup/pkg/migrate/migrate.go
package migrate

import (
    "fmt"

    "github.com/golang-migrate/migrate/v4"
    _ "github.com/golang-migrate/migrate/v4/source/file"
)

// Schema versions for DarkPipe configuration
const (
    SchemaV1 = "1"  // Initial release
    SchemaV2 = "2"  // Added Docker secrets support
    SchemaV3 = "3"  // Added component selection
)

func MigrateConfig(configPath string, targetVersion string) error {
    m, err := migrate.New(
        "file://migrations",
        "darkpipe://"+configPath,
    )
    if err != nil {
        return fmt.Errorf("migration init failed: %w", err)
    }
    defer m.Close()

    // Get current version
    version, dirty, err := m.Version()
    if err != nil && err != migrate.ErrNilVersion {
        return err
    }

    if dirty {
        return fmt.Errorf("config is in dirty state (version %d), manual intervention required", version)
    }

    // Migrate to target version
    if targetVersion == "latest" {
        if err := m.Up(); err != nil && err != migrate.ErrNoChange {
            return err
        }
    } else {
        target, _ := parseVersion(targetVersion)
        if err := m.Migrate(target); err != nil && err != migrate.ErrNoChange {
            return err
        }
    }

    return nil
}

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Separate `docker build` per arch | Buildx with `--platform` flag | Docker Buildx v0.4+ (2020) | Single command builds all architectures simultaneously with shared layer cache |
| Personal access tokens for GHCR | GITHUB_TOKEN in Actions | GitHub Actions 2021+ | No need to store secrets; automatic permissions via workflow token |
| Environment variables for secrets | Docker secrets with _FILE suffix | Docker Compose 3.1+ (2017) | Secrets never exposed in env vars or logs; filesystem permissions control access |
| Static version tags | Semantic versioning with metadata | OCI Image Spec 1.0 (2017) | Automated version inference from git tags, immutable SHA tags for reproducibility |
| Manual ARM64 builds | QEMU emulation in CI | GitHub Actions + QEMU (2020+) | No need for native ARM runners for most builds (use sparingly for releases) |
| Custom Dockerfile per architecture | Multi-platform Dockerfile with TARGETARCH | BuildKit 0.4+ (2020) | Single Dockerfile with `--platform` buildx automatically sets TARGETARCH/TARGETOS |

**Deprecated/outdated:**

- **Docker Compose v2 (v1.x CLI):** Deprecated in favor of `docker compose` (v2 plugin). Migration: Replace `docker-compose` with `docker compose` in all docs/scripts. (Changed: 2023, Impact: v1 no longer receives updates)

- **Docker Hub as default registry:** GHCR (ghcr.io) now preferred for GitHub-hosted projects due to better integration, unlimited public image storage, and higher rate limits. (Changed: 2020+, Impact: Simpler auth with GITHUB_TOKEN, no separate Docker Hub account needed)

- **Mutable 'latest' tag for production:** Industry now prefers immutable tags (SHA, semver) with 'latest' reserved for development/testing. (Changed: ~2022, Impact: Reproducible deployments, easier rollback)

- **UPX compression by default:** While UPX can reduce binary size 70-80%, the ~160ms startup overhead and potential compatibility issues mean it's now opt-in rather than default. (Changed: 2024+, Impact: Faster container startup, fewer edge-case bugs)

## Open Questions

### 1. Default Stack Selection

**What we know:**
- User research shows Stalwart (modern, all-in-one, Rust) vs Maddy (minimal, Go) vs Postfix+Dovecot (battle-tested) trade-offs
- Existing STACK.md recommends Stalwart for built-in CalDAV/CardDAV and JMAP support
- SnappyMail has better performance than Roundcube (138KB vs multi-MB page loads)

**What's unclear:**
- Should pre-built image use Stalwart + SnappyMail (modern), Stalwart + Roundcube (familiar), or Postfix+Dovecot + Roundcube (most conservative)?
- Is "opinionated default" literally one combo, or "recommended" with explanation of alternatives?

**Recommendation:** Default to **Stalwart + SnappyMail** (most modern, smallest footprint, built-in calendar). Provide **Postfix+Dovecot + Roundcube** as "conservative" pre-built alternative. Document reasoning clearly: "We recommend Stalwart for new deployments; choose Postfix+Dovecot if you need maximum compatibility or have existing expertise."

### 2. Image Architecture: Monolithic vs Separate

**What we know:**
- Separate images per component allow mix-and-match stacks and smaller per-component size
- Monolithic images simpler to deploy (single pull, coordinated versioning)
- Current docker-compose.yml uses separate images (stalwart, maddy, postfix-dovecot, roundcube, snappymail, radicale)

**What's unclear:**
- Should we continue separate images, or offer monolithic "home-device-all" image?
- How to version separate images (same tag across all components, or independent)?

**Recommendation:** Continue **separate images per component** to maintain flexibility and minimize size. Use **unified versioning** (all components share same git tag/SHA) for coordinated releases, but allow users to mix versions if they customize. Monolithic image adds complexity (conditional component startup in entrypoint) without clear deployment benefit since users already run `docker compose up`.

### 3. Health Check Scope

**What we know:**
- Phase 9 covers comprehensive monitoring (mail queue depth, delivery status, certificate expiry)
- Basic health checks needed in Phase 7 for container orchestration
- Existing Dockerfiles have minimal health checks (e.g., `postfix status`)

**What's unclear:**
- Should Phase 7 bundle basic `darkpipe doctor` command (check DNS, test SMTP, verify certs), or defer all diagnostics to Phase 9?
- What's the minimum viable health check for Phase 7 container orchestration?

**Recommendation:** Phase 7 should include **basic container-level health checks only** (process running, ports listening, essential dependencies accessible). Example: `HEALTHCHECK CMD nc -z localhost 25 && nc -z localhost 587 || exit 1`. Defer comprehensive diagnostics to Phase 9 `darkpipe doctor` command. This keeps Phase 7 focused on deployment and avoids scope creep.

### 4. RISC-V Feasibility

**What we know:**
- Docker supports riscv64 architecture, base images available (Debian, Ubuntu)
- Some Kubernetes tools (k0s) support RISC-V but no prebuilt binaries yet
- RISC-V hardware still emerging (limited availability, mostly dev boards)

**What's unclear:**
- Is RISC-V support worth investing time in Phase 7 given minimal user base?
- What's the build time impact (QEMU for riscv64 likely 10x slower than arm64)?

**Recommendation:** **Defer RISC-V to post-v1** (v2 "additional architectures" stretch goal). Rationale: Minimal user impact, unclear hardware adoption timeline, significant build time cost for unproven demand. Document as "under evaluation" to gauge community interest. Re-evaluate if RISC-V SBC market matures (e.g., widespread Raspberry Pi RISC-V equivalent).
