<objective>
Set up the home device mail server foundation with all three selectable options (Stalwart, Maddy, Postfix+Dovecot), each providing inbound SMTP (port 25), IMAP (port 993), and SMTP submission (port 587).

Purpose: Establishes the receiving end of the mail pipeline -- the cloud relay (Phase 2) forwards mail via SMTP to port 25 on the home device, and this plan ensures a working mail server is listening there, storing messages, and serving them to IMAP clients. Docker compose profiles enable runtime mail server selection per MAIL-01.

Output: Complete home-device/ directory with Dockerfiles, config files, and docker-compose.yml for all three mail server options.
</objective>

Create the home-device/ directory structure following the research-recommended layout.

**home-device/docker-compose.yml:**
- Use Docker Compose v3.8 format (consistent with cloud-relay/docker-compose.yml)
- Define three services with profiles for selection:
  - `stalwart` service with profile `["stalwart"]` -- uses stalwartlabs/stalwart:0.15.4 image (do NOT build custom, use official multi-arch image)
  - `maddy` service with profile `["maddy"]` -- uses foxcpp/maddy:0.8.2 image
  - `postfix-dovecot` service with profile `["postfix-dovecot"]` -- builds from local Dockerfile (no official combined image)
- All three expose identical ports: 25 (SMTP inbound from relay), 587 (submission), 993 (IMAPS)
- Shared `darkpipe` network (bridge) for inter-container communication
- Named volumes for mail storage persistence:
  - `mail-data` for mailbox storage
  - `mail-config` for configuration persistence
- Environment variables documented in comments: MAIL_DOMAIN, MAIL_HOSTNAME, ADMIN_EMAIL, ADMIN_PASSWORD
- Resource limits: memory 512M (home device has more resources than cloud relay)
- Usage: `docker compose --profile stalwart up -d` to select a mail server

**home-device/stalwart/Dockerfile:**
- FROM stalwartlabs/stalwart:0.15.4 (official multi-arch, supports arm64+amd64)
- COPY config.toml to /opt/stalwart-mail/etc/config.toml
- EXPOSE 25 587 993
- Use default CMD from base image (stalwart-mail binary)

**home-device/stalwart/config.toml:**
- Listener "smtp" on 0.0.0.0:25 (inbound from cloud relay, no auth required for local delivery)
- Listener "submission" on 0.0.0.0:587 (STARTTLS, requires auth)
- Listener "imap" on 0.0.0.0:993 (implicit TLS)
- Internal directory with SQLite store for user accounts
- Storage: data=sqlite at /opt/stalwart-mail/data/index.db, blob=fs at /opt/stalwart-mail/data/blobs
- TLS: Use self-signed certificate initially (home device behind WireGuard, not public internet). Generate self-signed cert in Dockerfile or use Stalwart's built-in auto-generation. Document that users with public DNS should replace with Let's Encrypt.
- Session configuration: relay for local domains only, max message size 50MB (matches cloud relay RELAY_MAX_MESSAGE_BYTES)
- Logging to stdout for container observability

**home-device/maddy/Dockerfile:**
- FROM foxcpp/maddy:0.8.2 (official image)
- COPY maddy.conf to /etc/maddy/maddy.conf
- EXPOSE 25 587 993
- Use default CMD from base image

**home-device/maddy/maddy.conf:**
- Define $(hostname) and $(local_domains) variables (substituted from env or hardcoded with example values)
- SMTP endpoint on 0.0.0.0:25 for inbound delivery (no auth for local domains)
- Submission endpoint on 0.0.0.0:587 with TLS and auth
- IMAP endpoint on 0.0.0.0:993 with TLS and auth
- storage.imapsql with SQLite driver for mailbox storage
- auth.pass_table with shadow backend or file-based for user credentials
- TLS: self-signed certificate initially (same approach as Stalwart)
- Destination routing: local domains deliver to local_mailboxes, reject everything else (no open relay)

**Critical details:**
- Do NOT make any mail server an open relay. Port 25 MUST only accept mail for configured local domains.
- Stalwart and Maddy configs should be functionally equivalent -- same ports, same basic behavior, same security posture.
- Self-signed TLS certs are acceptable because IMAP/submission traffic is already inside the WireGuard tunnel. Document this clearly in config comments.
- Use environment variable substitution where the mail server supports it (Stalwart uses %{env:VAR}%, Maddy uses $(var) from command line or config).

Create the Postfix+Dovecot combined mail server option.

**home-device/postfix-dovecot/Dockerfile:**
- FROM alpine:3.21 (consistent with cloud relay)
- Install: postfix, dovecot, dovecot-lmtpd, dovecot-pigeonhole (sieve for future use), ca-certificates
- Use LMDB for Postfix maps (NOT BerkleyDB -- deprecated in Alpine, per decision from Phase 2)
- Create vmail user/group (UID/GID 5000) for virtual mailbox ownership
- Create /var/mail/vhosts directory for virtual mailbox storage
- COPY postfix/ and dovecot/ config directories
- COPY entrypoint.sh
- EXPOSE 25 587 993
- CMD ["/entrypoint.sh"]

**home-device/postfix-dovecot/postfix/main.cf:**
- myhostname = mail.example.com (placeholder, document envsubst)
- mydestination = (empty -- all delivery via virtual mailbox domains, not local)
- virtual_mailbox_domains = example.com (placeholder)
- virtual_mailbox_base = /var/mail/vhosts
- virtual_mailbox_maps = lmdb:/etc/postfix/vmailbox
- virtual_alias_maps = lmdb:/etc/postfix/virtual (for aliases, configured in Plan 02)
- virtual_uid_maps = static:5000
- virtual_gid_maps = static:5000
- virtual_transport = lmtp:unix:private/dovecot-lmtp (Postfix delivers to Dovecot via LMTP)
- smtpd_tls_cert_file, smtpd_tls_key_file pointing to /etc/ssl/mail/ (self-signed initially)
- smtpd_tls_security_level = may (opportunistic within WireGuard tunnel)
- smtpd_sasl_auth_enable = yes (for submission port)
- smtpd_sasl_type = dovecot
- smtpd_sasl_path = private/auth
- smtpd_relay_restrictions = permit_sasl_authenticated, reject_unauth_destination (prevents open relay)
- message_size_limit = 52428800 (50MB, matches cloud relay)
- Use LMDB format for ALL map files (lmdb: prefix, not hash:)
- smtpd_recipient_restrictions: permit_mynetworks (WireGuard subnet 10.8.0.0/24 for inbound from relay), permit_sasl_authenticated, reject_unauth_destination
- mynetworks = 127.0.0.0/8, 10.8.0.0/24 (loopback + WireGuard subnet)
- Logging to stdout for container

**home-device/postfix-dovecot/postfix/master.cf:**
- smtp/25: standard smtpd (accepts inbound from cloud relay)
- submission/587: smtpd with -o smtpd_sasl_auth_enable=yes -o smtpd_tls_security_level=encrypt -o smtpd_recipient_restrictions=permit_sasl_authenticated,reject (authenticated submission only)

**home-device/postfix-dovecot/dovecot/dovecot.conf:**
- protocols = imap lmtp
- listen = *
- ssl = required (for IMAP)
- ssl_cert, ssl_key pointing to /etc/ssl/mail/ (self-signed initially)
- ssl_min_protocol = TLSv1.2
- mail_location = maildir:/var/mail/vhosts/%d/%n/Maildir
- passdb with passwd-file driver, args = /etc/dovecot/users (simple file-based auth for Phase 3, sufficient for multi-user)
- userdb with static driver, args = uid=5000 gid=5000 home=/var/mail/vhosts/%d/%n
- service lmtp: unix_listener /var/spool/postfix/private/dovecot-lmtp with user=postfix, group=postfix, mode=0600
- service auth: unix_listener /var/spool/postfix/private/auth with user=postfix, group=postfix, mode=0660
- namespace inbox with inbox=yes, separator=/
- auth_mechanisms = plain login (within TLS, these are safe)
- disable_plaintext_auth = yes (force TLS for IMAP auth)

**home-device/postfix-dovecot/entrypoint.sh:**
- Generate self-signed TLS certificate if none exists at /etc/ssl/mail/
- Use openssl req -x509 -nodes -days 3650 -newkey rsa:2048 for self-signed cert generation
- Create initial empty map files: /etc/postfix/vmailbox, /etc/postfix/virtual
- Run postmap on all map files (lmdb format)
- Create default admin user in /etc/dovecot/users if file doesn't exist (format: user@domain:{PLAIN}password -- document that user must change)
- Start Dovecot in background
- Start Postfix in foreground (exec postfix start-fg)
- Trap SIGTERM for graceful shutdown of both services

**Critical details:**
- LMTP is correct here (Postfix->Dovecot on same host, NOT from cloud relay). This is internal delivery, not the cloud relay -> home path which uses SMTP to port 25.
- The flow is: Cloud relay SMTP -> Postfix port 25 -> Postfix virtual_transport -> Dovecot LMTP -> maildir
- Postfix and Dovecot communicate via Unix sockets (LMTP + SASL auth), not TCP
- vmail user (UID 5000) owns all mailbox files -- no system users needed
- chmod 0755 on entrypoint.sh
