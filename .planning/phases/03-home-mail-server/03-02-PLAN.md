**Stalwart aliases and catch-all:**
- Add alias configuration to config.toml using Stalwart's built-in alias mechanism
- Aliases are managed via REST API: POST /api/v1/account with type=alias, email=alias@domain, memberOf=target@domain
- Document alias management in setup-users.sh (add curl commands for creating aliases)
- Add catch-all configuration: use Stalwart's catch-all rule that redirects unmatched addresses to a designated account
- Example aliases: admin@example.com -> alice@example.com, support@example.com -> alice@example.com
- Example catch-all: *@example.org -> bob@example.org
- Add comment warning: catch-all without spam filtering increases spam load (Plan 03 addresses this)

**Maddy aliases and catch-all:**
- Create aliases file at /etc/maddy/aliases with format: alias@domain: target@domain
- Update maddy.conf to reference the aliases file using table.file directive
- Add alias resolution in the delivery pipeline (modify section for destination local_domains)
- Configure catch-all: use Maddy's destination modifiers with replace_rcpt for unmatched addresses
- Example entries in aliases file:
  ```
  admin@example.com: alice@example.com
  support@example.com: alice@example.com
  ```
- Catch-all: Add catch-all mapping for example.org domain to bob@example.org
- Update Dockerfile to COPY aliases file

**Postfix+Dovecot aliases and catch-all:**
- Populate the virtual alias file (/etc/postfix/virtual) with:
  ```
  # Aliases within virtual mailbox domains
  admin@example.com    alice@example.com
  support@example.com  alice@example.com

  # Catch-all for example.org (all undefined addresses go to bob)
  @example.org         bob@example.org
  ```
- main.cf already has virtual_alias_maps = lmdb:/etc/postfix/virtual (set in Plan 01)
- Update entrypoint.sh to run `postmap lmdb:/etc/postfix/virtual` at startup
- Add comment in virtual file: "WARNING: Catch-all accepts ALL mail for domain including spam. Enable Rspamd before using catch-all in production."

**Critical anti-patterns to avoid (from research):**
1. Do NOT put catch-all domains in virtual_alias_domains -- they MUST stay in virtual_mailbox_domains. The catch-all entry (@example.org) in virtual_alias_maps works within virtual_mailbox_domains. Per research Pitfall 2.
2. Do NOT enable catch-all without documenting the spam risk. Plan 03 (Rspamd) should be deployed before catch-all is active in production. Per research Pitfall 3.
3. Aliases are in virtual_alias_maps; mailboxes are in virtual_mailbox_maps. These are separate files with separate purposes.

**Documentation in config comments:**
- Each alias file should include examples of single alias, multi-target alias, and catch-all
- Include warning about catch-all spam implications
- Document how to add new aliases (different for each mail server option)
