    **Dockerfile — Profile server container:**
    - Multi-stage build: `golang:1.24-alpine` builder → `alpine:3.21` runtime.
    - `COPY go.mod go.sum ./` then `go mod download` for layer caching.
    - `COPY . .` then `CGO_ENABLED=0 go build -ldflags="-s -w" -o /profile-server ./cmd/profile-server/`.
    - Runtime stage: Copy binary, templates, and static assets. Set USER nonroot. Expose port 8090.
    - Use ARG TARGETARCH for multi-arch support (matches Phase 7 pattern).
    - OCI labels: org.opencontainers.image.source, description, licenses=AGPL-3.0.
    - HEALTHCHECK: `wget --quiet --tries=1 --spider http://localhost:8090/health || exit 1`.

    **home-device/docker-compose.yml — Add profile-server service:**
    - New service `profile-server` (no profile needed -- runs with ALL mail server configurations):
      ```yaml
      profile-server:
        build:
          context: ./profiles
          dockerfile: Dockerfile
        container_name: profile-server
        restart: unless-stopped
        ports:
          - "8090:8090"
        environment:
          - PROFILE_SERVER_PORT=8090
          - MAIL_DOMAIN=${MAIL_DOMAIN:-example.com}
          - MAIL_HOSTNAME=${MAIL_HOSTNAME:-mail.example.com}
          - MAIL_SERVER_TYPE=${MAIL_SERVER_TYPE:-stalwart}
          - CALDAV_URL=${CALDAV_URL:-}
          - CARDDAV_URL=${CARDDAV_URL:-}
          - ADMIN_USER=${ADMIN_EMAIL:-admin@example.com}
          - ADMIN_PASSWORD=${ADMIN_PASSWORD:-changeme}
          - STALWART_ADMIN_USER=${ADMIN_EMAIL:-admin@example.com}
          - STALWART_ADMIN_PASSWORD=${ADMIN_PASSWORD:-changeme}
        volumes:
          - profile-data:/data
        networks:
          - darkpipe
        deploy:
          resources:
            limits:
              memory: 64M
        healthcheck:
          test: ["CMD", "wget", "--quiet", "--tries=1", "--spider", "http://localhost:8090/health", "||", "exit", "1"]
          interval: 30s
          timeout: 10s
          retries: 3
      ```
    - Add `profile-data` to volumes section.
    - Service runs WITHOUT a profile (always available, like rspamd/redis).

    **cloud-relay/docker-compose.yml — Add autoconfig/autodiscover env vars to Caddy:**
    - Add to caddy service environment:
      ```yaml

      AUTODISCOVER_DOMAINS: autodiscover.${RELAY_DOMAIN:-example.com}
      ```
    - Keep all existing environment variables and configuration.

    **webui.go — Web UI handlers for device management:**
    - Per user decision: "Add Device" button in webmail UI. Implementation via iframe approach (research recommendation): profile server serves its own HTML pages, webmail embeds via iframe or direct link.

    `WebUIHandler` struct with these HTTP routes:
    - `GET /devices` — Device management page. Lists all devices (app passwords) for the authenticated user. Shows device name, created date, last used date, and revoke button. Requires Basic Auth (same credentials as mail account).
    - `GET /devices/add` — "Add Device" page with:
      1. Device name input field (e.g., "iPhone 15", "Work Laptop")
      2. Platform selector: iOS/macOS, Android, Thunderbird, Outlook, Other
      3. On submit: generates app password + appropriate profile/instructions
      4. For iOS/macOS: Shows download .mobileconfig button + QR code
      5. For Android: Shows QR code + manual settings instructions
      6. For Thunderbird/Outlook: Shows "just enter your email and password" + app password
      7. For Other: Shows manual IMAP/SMTP settings + app password
    - `POST /devices/add` — Process add device form. Creates app password, stores via apppassword.Store, returns result page with password displayed ONCE (never shown again after this page).
    - `POST /devices/revoke` — Revoke app password by ID. Requires confirmation. Redirects back to /devices.

    HTML templates use Go html/template (embedded via embed.FS):
    - Clean, minimal design. No JavaScript frameworks. Mobile-responsive CSS.
    - DarkPipe branding (simple logo/name in header).
    - Device list: table with device name, email, created date, last used, revoke button.
    - Add device: form with device name, platform selection, clear instructions per platform.
    - Success page: shows app password in large monospace font, QR code (for iOS/Android), download button (for iOS), "you will not see this password again" warning.

    **cli.go — CLI QR code command:**
    - Per user decision: `darkpipe qr user@domain` generates and displays QR code in terminal.
    - When profile-server binary is invoked with `qr` subcommand, generates QR code for the given email:
      1. Create single-use token via token store
      2. Generate QR code URL: `https://<mail-hostname>/profile/download?token=<token>`
      3. Display QR code as ASCII art in terminal using go-qrcode ToString
      4. Print the URL below the QR code
      5. Print token expiry time (15 minutes)
      6. Print instructions: "Scan with your phone camera to install mail profile"
    - Also support `--png <file>` flag to save QR code as PNG file instead of terminal display.
    - Uses same token store and profile generation as the HTTP server.
    - Connect to profile server via HTTP API for token creation (or run in standalone mode with direct token store access if PROFILE_SERVER_URL not set).

    **static/style.css — Minimal responsive CSS:**
    - Max-width: 600px centered container.
    - System font stack (no web fonts to load).
    - Mobile-responsive: works on phone screens.
    - Dark text on white background. Blue accent color for buttons/links.
    - Monospace font for app passwords and QR code URLs.
    - Simple table styling for device list.
    - Warning styling (red border) for "password shown once" and "revoke" confirmations.

    Create comprehensive integration test script following the Phase 3 test suite pattern (test-mail-flow.sh).

    **tests/test-device-profiles.sh:**

    ```bash
    #!/usr/bin/env bash
    # Phase 8: Device Profiles & Client Setup - Integration Test Suite
    # Tests all PROF-01 through PROF-05 requirements
    #
    # Prerequisites:
    #   - Profile server running on port 8090
    #   - Mail server running (any profile: stalwart, maddy, postfix-dovecot)
    #   - curl, jq available
    #
    # Usage:
    #   ./tests/test-device-profiles.sh [--profile-server-url URL] [--mail-domain DOMAIN]
    ```

    Test sections (each maps to a requirement):

    **PROF-01: Apple .mobileconfig profiles**
    - Generate a .mobileconfig via the profile download endpoint (create token first via API, then redeem).
    - Verify response Content-Type is `application/x-apple-aspen-config`.
    - Verify response contains `PayloadType` and `com.apple.mail.managed` (email payload).
    - Verify response contains CalDAV payload (`com.apple.caldav.account`) if CalDAV is configured.
    - Verify response contains CardDAV payload (`com.apple.carddav.account`) if CardDAV is configured.
    - If `plutil` available (macOS), verify profile is valid plist: `plutil -lint`.

    **PROF-02: Android/autoconfig profiles**
    - Fetch `/.well-known/autoconfig/mail/config-v1.1.xml` from profile server.
    - Verify response is valid XML.
    - Verify XML contains `<incomingServer type="imap">` with port 993 and SSL.
    - Verify XML contains `<outgoingServer type="smtp">` with port 587 and STARTTLS.
    - Verify `%EMAILADDRESS%` placeholder present for username.

    **PROF-03: QR code generation**
    - Call QR generation endpoint, verify PNG image returned (check magic bytes `\x89PNG`).
    - Verify QR code URL contains `/profile/download?token=`.
    - Test single-use enforcement: redeem token once (200), try again (401).
    - Test expired token (create token with 1-second expiry if test API supports it, or just verify 401 for invalid token).

    **PROF-04: Desktop autodiscovery**
    - Fetch `/autodiscover/autodiscover.xml` from profile server.
    - Verify valid XML response.
    - Verify IMAP settings (port 993, SSL).
    - Verify SMTP settings (port 587, TLS/STARTTLS).
    - Verify SRV records generated by DNS tool contain _imaps._tcp and _submission._tcp (run dns-setup --json if available, or just verify the Go test output).

    **PROF-05: App-generated passwords**
    - Create an app password via the API/web UI endpoint.
    - Verify password format matches XXXX-XXXX-XXXX-XXXX.
    - List app passwords for the user, verify the new one appears.
    - Revoke the app password, verify it no longer appears in the list.
    - Verify a revoked password cannot authenticate (if IMAP test possible: `openssl s_client` + IMAP LOGIN).

    **Output format:**
    - Each test prints `[PASS]` or `[FAIL]` with description.
    - Final summary: X/Y tests passed.
    - Exit code 0 if all pass, 1 if any fail.
    - Follow same pattern as tests/test-mail-flow.sh and tests/test-spam-filter.sh.

    Script should be executable (chmod +x) and have comprehensive error handling (set -euo pipefail at top, trap for cleanup).
