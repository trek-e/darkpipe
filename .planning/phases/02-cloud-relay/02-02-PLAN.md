Build the TLS monitoring, strict mode, and notification subsystems.

1. **cloud-relay/relay/notify/notifier.go** - Notification interface:
   - `Event` struct: `Type` (string: "tls_warning", "tls_failure"), `Domain` (remote server domain), `Message` (human-readable description), `Timestamp` (time.Time), `Details` (map[string]string for metadata like IP, port, TLS version attempted)
   - `Notifier` interface: `Send(ctx context.Context, event Event) error`, `Close() error`
   - `MultiNotifier` struct that dispatches to multiple backends (fan-out). Implements Notifier interface. Send() calls all backends, collects errors.
   - `NewMultiNotifier(notifiers ...Notifier) *MultiNotifier`

2. **cloud-relay/relay/notify/webhook.go** - Webhook notification backend:
   - `WebhookNotifier` struct with `URL` (webhook endpoint), `httpClient` (*http.Client with 10s timeout)
   - `NewWebhookNotifier(url string) *WebhookNotifier`
   - `Send()` POSTs JSON-encoded Event to webhook URL. Log failures but do not block mail flow (notifications are best-effort).
   - Include `X-DarkPipe-Event` header with event type
   - Rate limit: deduplicate notifications for same domain within 1 hour (use sync.Map with domain -> lastNotified timestamp). This prevents notification spam for domains that consistently fail TLS.

3. **cloud-relay/relay/notify/notifier_test.go** - Tests:
   - Test MultiNotifier dispatches to all backends
   - Test MultiNotifier collects errors from failing backends
   - Test WebhookNotifier rate limiting (same domain within 1 hour suppressed)

4. **cloud-relay/relay/tls/monitor.go** - Postfix log TLS monitor:
   - `TLSMonitor` struct with `notifier` (notify.Notifier), `logReader` (io.Reader)
   - `NewTLSMonitor(logReader io.Reader, notifier notify.Notifier) *TLSMonitor`
   - `Monitor(ctx context.Context) error` — reads lines from logReader, detects TLS-related patterns:
     - Pattern: `Anonymous TLS connection established` — log successful TLS (info level)
     - Pattern: `TLS is required, but was not offered` — emit "tls_failure" event with extracted domain
     - Pattern: `untrusted issuer` or `certificate verification failed` — emit "tls_warning" event
     - Pattern: `Cannot start TLS` or `TLS handshake failed` — emit "tls_warning" event
   - Extract domain from Postfix log line: parse `to=<user@domain>` or connection info
   - Use bufio.Scanner for line-by-line reading. When ctx cancelled, return nil.
   - IMPORTANT: This reads from a pipe/stream (not a static file). In production, it reads from Postfix's maillog output (piped via `postfix start-fg` stdout or a named pipe).

5. **cloud-relay/relay/tls/monitor_test.go** - Tests:
   - Test detection of "TLS is required, but was not offered" pattern
   - Test detection of "TLS handshake failed" pattern
   - Test domain extraction from Postfix log lines
   - Test that successful TLS connections do NOT trigger notifications
   - Test that context cancellation stops monitor cleanly

6. **cloud-relay/relay/tls/strict.go** - Strict mode configuration:
   - `StrictMode` struct with `Enabled` (bool), `RefuseAllPlaintext` (bool — refuse ALL plaintext, not per-domain), `PolicyMapPath` (default "/etc/postfix/tls_policy")
   - `GeneratePolicyMap()` — if RefuseAllPlaintext, writes a catch-all `* encrypt` rule to policy map file. Runs `postmap lmdb:<path>` to hash it.
   - `ApplyToPostfix()` — updates Postfix main.cf by setting `smtp_tls_security_level = encrypt` (for outbound) and `smtpd_tls_security_level = encrypt` (for inbound) when strict mode enabled. Uses `postconf -e` command to apply settings dynamically (no file editing needed, Postfix reloads).
   - `DisableStrictMode()` — reverts to `may` security level using `postconf -e`
   - Note: inbound strict mode (`smtpd_tls_security_level = encrypt`) means remote servers MUST use STARTTLS. This will reject mail from ancient servers that cannot do TLS. This is the user's conscious choice when enabling strict mode.

7. **cloud-relay/relay/tls/strict_test.go** - Tests:
   - Test policy map generation with strict enabled
   - Test policy map generation with strict disabled (no file created)
   - Test postconf command construction (mock execution)

8. **Update cloud-relay/relay/config/config.go** — Add fields:
   - `StrictModeEnabled` (bool, env: `RELAY_STRICT_MODE`, default: false)
   - `WebhookURL` (string, env: `RELAY_WEBHOOK_URL`, default: empty — notifications disabled if empty)

9. **Update cloud-relay/cmd/relay/main.go** — Wire up:

   - If StrictModeEnabled, apply strict mode configuration at startup
   - Ensure graceful shutdown closes monitor and notifier

Set up Let's Encrypt certificate automation using a certbot sidecar container.

1. **cloud-relay/certbot/docker-compose.certbot.yml** - Certbot sidecar compose overlay:
   - `certbot` service using `certbot/certbot:latest` image
   - Volume mounts: `certbot-etc:/etc/letsencrypt`, `certbot-var:/var/lib/letsencrypt`
   - Entrypoint script that:
     a. On first run: obtains certificate via `certbot certonly --standalone --agree-tos --email ${CERTBOT_EMAIL} -d ${RELAY_HOSTNAME} --non-interactive`
     b. If cert already exists: skips initial obtain
     c. Enters renewal loop: `while true; do certbot renew --deploy-hook /etc/certbot/renew-hook.sh; sleep 12h; done`
   - Environment vars: `CERTBOT_EMAIL`, `RELAY_HOSTNAME`
   - NOTE: Uses standalone mode (port 80) for HTTP-01 challenge during initial obtain. After that, renewal uses the existing cert. Document that port 80 must be briefly available during initial setup, OR user can use DNS-01 challenge by overriding certbot command.
   - Expose port 80 temporarily for initial certificate obtain only
   - For production, document DNS-01 challenge alternative: `certbot certonly --dns-cloudflare --dns-cloudflare-credentials /etc/cloudflare.ini -d ${RELAY_HOSTNAME}`

2. **cloud-relay/certbot/renew-hook.sh** - Post-renewal hook:
   - `#!/bin/sh`
   - `postfix reload` (signals Postfix to re-read TLS certificates without restarting)
   - This runs inside the certbot container but needs access to the relay container's Postfix. Two approaches:
     a. Shared volume: certbot writes certs to shared volume, relay's entrypoint watches for cert changes (inotifywait or periodic check)
     b. Docker socket: certbot sends docker exec to relay container
   - Use approach (a): certbot writes to shared volume. Add a cert-watcher loop in entrypoint.sh that checks cert modification time every 5 minutes and runs `postfix reload` if changed. This avoids Docker socket security concerns.

3. **Update cloud-relay/docker-compose.yml** — Integrate certbot:
   - Add `certbot-etc` and `certbot-var` named volumes
   - Mount `certbot-etc:/etc/letsencrypt:ro` in relay service
   - Add certbot service (or reference overlay file)
   - Add environment variables: `CERTBOT_EMAIL`, `RELAY_HOSTNAME`
   - Expose port 80 on certbot service for HTTP-01 challenge

4. **Update cloud-relay/entrypoint.sh** — Add cert watcher:
   - After starting Postfix, start a background loop that:
     a. Every 300 seconds (5 min), checks if `/etc/letsencrypt/live/${RELAY_HOSTNAME}/fullchain.pem` mtime has changed
     b. If changed, runs `postfix reload` to pick up new certs
     c. Logs cert reload events
   - If TLS certs do not exist yet (first boot before certbot runs), configure Postfix with `smtpd_tls_security_level = none` temporarily and log a warning. Switch to `may` once certs appear.
   - Add `RELAY_STRICT_MODE` and `RELAY_WEBHOOK_URL` env var forwarding to relay daemon

5. **Update cloud-relay/postfix-config/main.cf** — Ensure TLS configuration is complete:
   - Verify `smtpd_tls_cert_file` and `smtpd_tls_key_file` point to Let's Encrypt cert paths
   - Add `smtpd_tls_received_header = yes` (adds TLS info to Received headers for debugging)
   - Add `tls_preempt_cipherlist = yes` (server chooses cipher preference)
   - Ensure `smtp_tls_session_cache_database` and `smtpd_tls_session_cache_database` use `lmdb:` prefix
