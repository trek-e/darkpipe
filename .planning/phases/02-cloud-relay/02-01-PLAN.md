<objective>
Build the core cloud relay: a Postfix relay-only container with a Go SMTP relay daemon that bridges internet-facing SMTP to the home device via WireGuard or mTLS transport.

Purpose: This is the foundational plan for Phase 2. It establishes the inbound mail flow (internet -> Postfix -> Go daemon -> transport -> home device) and outbound mail flow (home device -> transport -> Postfix -> direct MTA delivery). Without this, no mail flows through the system.

Output: A working Docker container running Postfix in relay-only mode and a Go relay daemon that forwards mail to the home device via Phase 1's transport layer.
</objective>

  <action>
Create the Go relay daemon under cloud-relay/ directory within the existing github.com/darkpipe/darkpipe module.

1. **cloud-relay/relay/config/config.go** - Configuration struct:
   - `Config` struct with fields: `ListenAddr` (default "127.0.0.1:10025"), `TransportType` (wireguard|mtls), `HomeDeviceAddr` (host:port of home mail server, default "10.8.0.2:25" for WireGuard, configurable for mTLS), `CACertPath`, `ClientCertPath`, `ClientKeyPath` (for mTLS), `MaxMessageBytes` (default 50MB), `ReadTimeout`/`WriteTimeout` (default 30s)
   - `LoadFromEnv()` function that reads `RELAY_LISTEN_ADDR`, `RELAY_TRANSPORT`, `RELAY_HOME_ADDR`, `RELAY_CA_CERT`, `RELAY_CLIENT_CERT`, `RELAY_CLIENT_KEY`, `RELAY_MAX_MESSAGE_BYTES` environment variables with sensible defaults
   - Validate required fields based on transport type (mTLS needs cert paths, WireGuard needs home addr only)

2. **cloud-relay/relay/forward/forwarder.go** - Transport abstraction:
   - `Forwarder` interface with `Forward(ctx context.Context, from string, to []string, data io.Reader) error` and `Close() error`
   - This is the bridge between SMTP session and transport layer

3. **cloud-relay/relay/forward/mtls.go** - mTLS forwarding:
   - `MTLSForwarder` struct wrapping `transport/mtls/client.Client` from Phase 1
   - `NewMTLSForwarder(caCert, clientCert, clientKey, homeAddr string) (*MTLSForwarder, error)` - creates Phase 1 mTLS client
   - `Forward()` implementation: connect via mTLS client.Connect(), then use emersion/go-smtp client to send SMTP envelope (MAIL FROM, RCPT TO, DATA) to home device's SMTP server over the mTLS connection
   - Use `github.com/emersion/go-smtp` client-side functions to properly send SMTP commands rather than raw byte copying

4. **cloud-relay/relay/forward/wireguard.go** - WireGuard forwarding:
   - `WireGuardForwarder` struct with `homeAddr` field (e.g., "10.8.0.2:25" via WireGuard tunnel)
   - `NewWireGuardForwarder(homeAddr string) *WireGuardForwarder`
   - `Forward()` implementation: dial `homeAddr` directly (WireGuard tunnel handles encryption transparently), then use go-smtp client to send SMTP envelope to home device's SMTP server
   - No special crypto needed — WireGuard kernel module handles encryption at network layer

5. **cloud-relay/relay/smtp/server.go** - SMTP backend:
   - `Backend` struct implementing `smtp.Backend` interface from emersion/go-smtp
   - Holds a `forward.Forwarder` reference
   - `NewSession()` creates a Session with the forwarder
   - `NewServer(forwarder forward.Forwarder, cfg *config.Config) *smtp.Server` factory function that configures the go-smtp server with ListenAddr, MaxMessageBytes, timeouts, domain name

6. **cloud-relay/relay/smtp/session.go** - SMTP session:
   - `Session` struct implementing `smtp.Session` interface
   - `Mail()` captures sender address
   - `Rcpt()` appends recipient addresses
   - `Data()` calls `forwarder.Forward(ctx, from, to, reader)` — this is the critical link that bridges SMTP to transport
   - `Reset()` clears session state
   - `Logout()` no-op (stateless)
   - Log each forwarding attempt with from/to/status for debugging

7. **cloud-relay/cmd/relay/main.go** - Entrypoint:
   - Load config via `config.LoadFromEnv()`
   - Create appropriate forwarder based on transport type (WireGuard or mTLS)
   - Create and start SMTP server
   - Handle SIGINT/SIGTERM for graceful shutdown with 30s timeout
   - Log startup with listen address and transport type

8. **go.mod/go.sum** - Add `github.com/emersion/go-smtp` dependency. Run `go mod tidy`.

Important: The Go relay daemon listens on localhost:10025 ONLY (not exposed to internet). Postfix handles internet-facing SMTP on port 25 and forwards to the daemon via transport_maps. Do NOT make the daemon listen on 0.0.0.0.

Create the Postfix relay-only configuration and Docker container setup.

1. **cloud-relay/postfix-config/main.cf** - Relay-only null client configuration:
   - `myhostname = ${RELAY_HOSTNAME}` (templated by entrypoint.sh)
   - `mydomain = ${RELAY_DOMAIN}` (templated by entrypoint.sh)
   - `myorigin = $mydomain`
   - `mydestination =` (empty — null client, NO local delivery)
   - `local_recipient_maps =` (empty)
   - `local_transport = error:local mail delivery is disabled`
   - `inet_interfaces = all`
   - `inet_protocols = ipv4` (IPv6 support deferred)
   - `smtpd_relay_restrictions = permit_mynetworks, reject_unauth_destination`
   - `mynetworks = 127.0.0.0/8, [::1]/128, 10.8.0.0/24` (include WireGuard subnet for home device outbound submission)
   - `transport_maps = hash:/etc/postfix/transport`
   - `message_size_limit = 52428800` (50MB)
   - `smtpd_tls_security_level = may` (STARTTLS offered but not required — Plan 02-02 adds enforcement)
   - `smtp_tls_security_level = may` (opportunistic TLS for outbound)
   - `smtp_tls_loglevel = 1` (log TLS connection info)
   - `smtpd_tls_loglevel = 1`
   - `smtpd_tls_cert_file = /etc/letsencrypt/live/${RELAY_HOSTNAME}/fullchain.pem` (placeholder — Plan 02-02 provides actual certs)
   - `smtpd_tls_key_file = /etc/letsencrypt/live/${RELAY_HOSTNAME}/privkey.pem`
   - `smtpd_tls_mandatory_protocols = !SSLv2, !SSLv3, !TLSv1, !TLSv1.1` (TLS 1.2+ only)
   - `smtp_tls_mandatory_protocols = !SSLv2, !SSLv3, !TLSv1, !TLSv1.1`
   - `smtp_tls_session_cache_database = lmdb:${data_directory}/smtp_scache` (use LMDB, not BerkleyDB per research)
   - `smtpd_tls_session_cache_database = lmdb:${data_directory}/smtpd_scache`
   - `maillog_file = /dev/stdout` (log to stdout for container observability)
   - `queue_directory = /var/spool/postfix` (persistent volume mount point)
   - IMPORTANT: Use `lmdb:` prefix for all database maps (not `hash:` which uses BerkleyDB on Alpine). Research explicitly warns: "BerkleyDB deprecated in Alpine; use LMDB"

2. **cloud-relay/postfix-config/master.cf** - Standard master.cf:
   - Standard smtp on port 25 (inet, n, -, -, -, smtpd)
   - No submission (587) or smtps (465) — home device submits through WireGuard tunnel, not via internet-facing ports
   - pickup, cleanup, qmgr, tlsmgr, rewrite, bounce, defer, trace, verify, flush, proxymap, proxywrite, smtp, relay, showq, error, retry, discard, local, virtual, lmtp, anvil, scache services (standard Postfix defaults)

3. **cloud-relay/postfix-config/transport** - Transport map:
   - Single wildcard rule: `* smtp:[127.0.0.1]:10025` — all inbound mail forwarded to Go relay daemon
   - Comment explaining: Postfix forwards to Go daemon, which forwards via WireGuard/mTLS to home device
   - NOTE: This file needs to be hashed with `postmap` in the entrypoint script. Use `postmap lmdb:/etc/postfix/transport` (LMDB format)

4. **cloud-relay/entrypoint.sh** - Container entrypoint:
   - Substitute environment variables in main.cf: `RELAY_HOSTNAME`, `RELAY_DOMAIN`
   - Use `envsubst` or `sed` to replace placeholders
   - Run `postmap lmdb:/etc/postfix/transport` to hash transport map
   - Run `postfix check` to validate configuration
   - Start relay daemon in background: `/usr/local/bin/relay-daemon &`
   - Start Postfix in foreground: `postfix start-fg`
   - Trap SIGTERM to gracefully stop both processes
   - Set permissions: ensure Postfix queue directory exists and is owned by postfix user

5. **cloud-relay/Dockerfile** - Multi-stage build:
   - Stage 1 (builder): `FROM golang:1.24-alpine AS builder`, build relay-daemon with `CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o relay-daemon ./cloud-relay/cmd/relay`
   - Stage 2 (runtime): `FROM alpine:3.21`, install `postfix ca-certificates wireguard-tools gettext` (gettext for envsubst)
   - Copy relay-daemon binary from builder
   - Copy postfix-config/ to /etc/postfix/
   - Copy entrypoint.sh, make executable
   - EXPOSE 25
   - HEALTHCHECK using `postfix status` with 30s interval
   - ENTRYPOINT ["/entrypoint.sh"]
   - Target: under 50MB total image size (Alpine 5MB + Postfix ~15MB + WireGuard tools ~5MB + Go binary ~5MB + deps ~10MB)

6. **cloud-relay/docker-compose.yml** - Development compose file:
   - Service `relay`: builds from Dockerfile, ports 25:25, environment vars for RELAY_HOSTNAME, RELAY_DOMAIN, RELAY_TRANSPORT, RELAY_HOME_ADDR
   - Volume mount for Postfix queue: `postfix-queue:/var/spool/postfix` (persistent to survive restarts per research recommendation)
   - Volume mount for TLS certs: `certbot-etc:/etc/letsencrypt:ro` (placeholder for Plan 02-02)
   - Network mode considerations: if using WireGuard, container needs NET_ADMIN capability and access to wg0 interface
   - Add `cap_add: [NET_ADMIN]` and `devices: [/dev/net/tun]` for WireGuard support
   - Named volumes at bottom

IMPORTANT: For the transport_maps directive in main.cf, use `lmdb:/etc/postfix/transport` NOT `hash:/etc/postfix/transport`. The research explicitly states BerkleyDB is deprecated in Alpine and LMDB is the default.
