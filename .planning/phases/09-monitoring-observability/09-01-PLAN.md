  <action>
    Create the monitoring/health/ package with deep readiness check architecture:

    **monitoring/health/checker.go:**
    - Define `type Checker struct` with configurable check functions
    - Define `type CheckResult struct` with Name, Status ("ok"/"error"), Message, Duration fields
    - Define `type HealthStatus struct` with Status ("up"/"down"), Checks []CheckResult, Timestamp
    - `Liveness() HealthStatus` - cheap check (process alive, returns immediately)
    - `Readiness() HealthStatus` - deep check (runs all registered checkers: Postfix, IMAP, tunnel)
    - `RegisterCheck(name string, fn CheckFunc)` for extensible check registration
    - JSON-serializable structs for API responses

    **monitoring/health/postfix.go:**
    - `CheckPostfix(ctx context.Context) CheckResult` - TCP dial to localhost:25 with 2-second timeout
    - Return "ok" if connection succeeds, "error" with dial error message if fails
    - Use net.DialTimeout (not full SMTP handshake, just port check for readiness)

    **monitoring/health/imap.go:**
    - `CheckIMAP(ctx context.Context) CheckResult` - TCP dial to localhost:993 with 2-second timeout
    - Same pattern as Postfix check but for IMAP port

    **monitoring/health/tunnel.go:**
    - `CheckTunnel(ctx context.Context) CheckResult` - leverage existing transport/health.Check()
    - Import and call transport/health.Check() with configured transport type
    - Convert transport HealthStatus to monitoring CheckResult
    - Support both WireGuard and mTLS based on TRANSPORT_TYPE env var

    **monitoring/health/server.go:**
    - HTTP handler functions (not a full server, just handlers for embedding)
    - `LivenessHandler(checker *Checker) http.HandlerFunc` -> GET /health/live returns 200 with JSON {"status":"up","timestamp":"..."}
    - `ReadinessHandler(checker *Checker) http.HandlerFunc` -> GET /health/ready returns 200 (healthy) or 503 (unhealthy) with full JSON status including all check results
    - Content-Type: application/health+json
    - Per user decision: deep readiness checks (actual service health, not just process alive)

    **monitoring/health/checker_test.go:**
    - Test Checker with mock check functions (pass and fail scenarios)
    - Test Liveness always returns up
    - Test Readiness aggregation (all pass = up, any fail = down)
    - Test HTTP handlers return correct status codes and JSON structure
    - Use httptest.NewRecorder for handler tests

    Create the monitoring/queue/ package for Postfix queue monitoring:

    **monitoring/queue/mailq.go:**
    - `type QueueMessage struct` with QueueID, QueueName, ArrivalTime (int64 unix epoch), MessageSize, Sender, Recipients fields
    - `type Recipient struct` with Address, DelayReason fields
    - `GetQueueStats() (*QueueStats, error)` - executes `postqueue -j`, parses NDJSON output
    - Parse each JSON line as QueueMessage struct
    - Note: Postfix postqueue -j outputs arrival_time as Unix epoch integer, NOT RFC3339 string
    - Handle empty queue (postqueue -j returns nothing) gracefully
    - 24-hour threshold for "stuck" message detection (configurable via StuckThreshold field)

    **monitoring/queue/stats.go:**
    - `type QueueStats struct` with Depth, Active, Deferred, Hold, Stuck int fields, OldestMessage time.Time, Timestamp time.Time
    - `type QueueSnapshot struct` embeds QueueStats plus Messages []QueueMessage for detailed view
    - `GetDetailedQueue() (*QueueSnapshot, error)` - returns full queue contents
    - JSON serializable for API and CLI consumption

    **monitoring/queue/mailq_test.go:**
    - Test JSON parsing with sample postqueue -j output (embedded test fixtures)
    - Test empty queue handling
    - Test stuck message detection (messages > 24h old)
    - Test deferred vs active vs hold queue name classification
    - Mock exec.Command using test helper (set PATH to include test binary or use interface)

    Create the monitoring/delivery/ package for mail delivery status tracking:

    **monitoring/delivery/parser.go:**
    - `type LogEntry struct` with Timestamp, QueueID, From, To, Status (delivered/deferred/bounced/sent), Relay, Delay, DSN, StatusDetail fields
    - `type Parser struct` - Postfix mail.log line parser
    - `ParseLine(line string) (*LogEntry, error)` - parse single Postfix log line
    - Regex patterns for Postfix log format:
      - `status=sent` -> delivered (outbound success)
      - `status=deferred` -> deferred (temporary failure)
      - `status=bounced` -> bounced (permanent failure)
      - `status=expired` -> expired (TTL exceeded)
      - Extract: to=<addr>, relay=host[ip]:port, delay=N, dsn=X.Y.Z, status=X (detail)
    - Track both inbound (from cloud relay to home) and outbound (from home to world) per Claude's discretion: both directions
    - Return nil for non-delivery log lines (connection, cleanup, etc.)

    **monitoring/delivery/tracker.go:**
    - `type DeliveryTracker struct` with ring buffer and mutex
    - `NewDeliveryTracker(maxEntries int)` - default 1000 entries per research recommendation
    - `Record(entry *LogEntry)` - add to ring buffer, thread-safe via sync.RWMutex
    - `GetRecent(n int) []LogEntry` - return last N entries (newest first)
    - `GetByQueueID(queueID string) []LogEntry` - lookup by Postfix queue ID
    - `GetStats() DeliveryStats` - aggregate counts (delivered, deferred, bounced) over tracker contents
    - Ring buffer implementation: use slice with write index, wrap around at maxEntries
    - System-wide tracking (not per-user) per Claude's discretion
    - 1000 message default, configurable via MONITOR_DELIVERY_HISTORY env var
    - In-memory only (no SQLite, per research anti-pattern avoidance)

    **monitoring/delivery/status.go:**
    - `type DeliveryStats struct` with Delivered, Deferred, Bounced, Expired, Total int, Period time.Duration (time span of oldest to newest)
    - `type DeliveryStatus struct` with Stats DeliveryStats, RecentEntries []LogEntry, Timestamp time.Time
    - `GetDeliveryStatus(tracker *DeliveryTracker, recentCount int) *DeliveryStatus` - convenience aggregator
    - JSON serializable for CLI and dashboard consumption

    **monitoring/delivery/parser_test.go:**
    - Test parsing real Postfix log line formats:
      - `"Feb 14 10:23:45 mail postfix/smtp[1234]: ABCDEF1234: to=<user@example.com>, relay=gmail-smtp-in.l.google.com[142.250.x.x]:25, delay=1.2, delays=0.1/0/0.5/0.6, dsn=2.0.0, status=sent (250 2.0.0 OK)"`
      - `"Feb 14 10:23:46 mail postfix/smtp[1234]: ABCDEF1235: to=<user@failing.com>, relay=none, delay=300, dsn=4.4.1, status=deferred (connect to failing.com[1.2.3.4]:25: Connection timed out)"`
      - `"Feb 14 10:23:47 mail postfix/smtp[1234]: ABCDEF1236: to=<user@invalid.com>, relay=none, delay=0, dsn=5.1.1, status=bounced (user unknown)"`
    - Test non-delivery lines return nil (connection established, cleanup, etc.)
    - Test edge cases: malformed lines, missing fields

    **monitoring/delivery/tracker_test.go:**
    - Test ring buffer wraps at capacity (add 1500 to 1000-capacity, oldest 500 dropped)
    - Test GetRecent returns newest first
    - Test GetByQueueID returns matching entries
    - Test GetStats aggregates correctly
    - Test thread safety: concurrent Record and GetRecent calls (use goroutines + sync.WaitGroup)
