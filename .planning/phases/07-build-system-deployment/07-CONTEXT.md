# Phase 7: Build System & Deployment - Context

**Gathered:** 2026-02-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Users produce custom Docker images tailored to their chosen stack components via GitHub Actions, with pre-built images available as an alternative, running on all target platforms. Includes an interactive setup script for first-run configuration and an upgrade-aware re-run path.

</domain>

<decisions>
## Implementation Decisions

### Component selection UX
- Opinionated default stack — one recommended combo so users can build without choosing
- Pre-built images available for users who want to skip CI entirely

### Image architecture
- Go glue daemon compiled into Docker images during build (not a separate binary)
- Aggressively optimize image size — Alpine base, multi-stage builds, strip binaries, target <100MB per image
- Publish to GHCR only (no Docker Hub)

### Deployment experience
- Interactive setup script written in Go (compiled binary, matches DarkPipe stack)
- Live validation during setup — check DNS, test ports, verify SMTP as user enters values
- Auto-detect missing setup on first `docker compose up` — print "run setup first" and exit gracefully
- Docker secrets support for sensitive values (passwords, API keys, DKIM private keys)
- Upgrade-aware — re-running setup detects existing config, preserves settings, migrates schema

### Platform targeting
- Primary targets: RPi4 (arm64), x64 Docker (generic), TrueNAS Scale, Unraid
- Additional targets: Proxmox LXC, Synology NAS, Mac Silicon (arm64 macOS)
- Evaluate: Other popular ARM boards, RISC-V feasibility
- Docker compose as universal deployment method
- Native app templates for TrueNAS Scale catalog and Unraid Community Applications

### Claude's Discretion
- Component selection mechanism (GitHub Actions inputs vs config file vs wizard)
- Default stack selection (which mail server + webmail combo to recommend)
- Pre-built image coverage (all combos vs default only vs most popular)
- Image architecture (monolithic vs separate images per component)
- Docker compose generation (template + .env vs generated by setup script)
- Health-check command scope (bundle basic `darkpipe doctor` vs defer to Phase 9)
- RPi4 resource enforcement (warn vs block vs document-only)
- Platform guide delivery (part of Phase 7 vs separate docs effort)

</decisions>

<specifics>
## Specific Ideas

- Docker secrets for sensitive values — user specifically requested this over plain .env for secrets
- Both docker-compose AND native app templates for TrueNAS/Unraid — not either/or
- Broad architecture support including Mac Silicon and ARM boards beyond RPi4
- RISC-V as an evaluation target (feasibility check, not hard commitment)

</specifics>
