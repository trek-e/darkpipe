# DarkPipe on Synology NAS

Deploy DarkPipe on Synology NAS using Container Manager (DSM 7.2+).

## Prerequisites

**Hardware:**
- Synology NAS (any model with DSM 7.2+ support)
- 4GB+ RAM (2GB minimum with optimization)
- 20GB+ free space on volume

**Software:**
- Synology DSM 7.2+ (required for Container Manager with Compose support)
- Container Manager package installed
- SSH access enabled (for advanced setup)

**Why DSM 7.2+?**
DSM 7.2 introduced native Docker Compose support via the Container Manager package. Earlier versions (DSM 6.x, DSM 7.0-7.1) use Docker package which has limited Compose support and may require manual CLI setup.

## Quick Start (Web UI Method)

1. **Package Center** → Install **Container Manager**
2. **Container Manager** → **Project** → **Create**
3. Upload `docker-compose.yml` generated by darkpipe-setup
4. **Build** and **Start**

## Detailed Steps

### Method 1: Container Manager UI (Recommended)

**Step 1: Install Container Manager**

1. Open **Package Center**
2. Search for **Container Manager**
3. Click **Install**
4. Wait for installation to complete

**Step 2: Create Shared Folder for DarkPipe**

1. Navigate to **Control Panel** → **Shared Folder**
2. Click **Create**
3. **Name**: `darkpipe`
4. **Location**: Choose volume with sufficient space
5. **Enable Recycle Bin**: ✓ (recommended for safety)
6. Click **OK**

**Step 3: Generate Docker Compose File**

**Option A: Via SSH (if enabled):**
```bash
# SSH into Synology
ssh admin@synology.local

# Create directory
mkdir -p /volume1/darkpipe
cd /volume1/darkpipe

# Download setup tool (x64 for most Synology NAS models)
curl -LO https://github.com/trek-e/darkpipe/releases/latest/download/darkpipe-setup-linux-amd64
chmod +x darkpipe-setup-linux-amd64

# Run setup
./darkpipe-setup-linux-amd64 setup
```

**Option B: Via File Station:**
1. Download `darkpipe-setup-linux-amd64` to your computer
2. Use **File Station** to upload to `darkpipe` shared folder
3. Use SSH to run it (or skip to Method 2 for manual setup)

**Step 4: Deploy Project via Container Manager**

1. Open **Container Manager**
2. Navigate to **Project** tab
3. Click **Create**
4. **Project Name**: `darkpipe`
5. **Path**: `/volume1/darkpipe` (or your shared folder path)
6. **Source**: Upload `docker-compose.yml` generated by setup tool
7. Click **Build**
8. Wait for images to pull (may take 5-10 minutes)
9. Click **Start**

**Step 5: Verify Deployment**

1. In **Container Manager** → **Container** tab
2. Verify all DarkPipe containers are **Running**
3. Click container name → **Logs** to view logs
4. Click container name → **Terminal** to access shell

### Method 2: SSH CLI (Advanced)

For users who prefer command-line control:

**Step 1: Enable SSH**

1. Navigate to **Control Panel** → **Terminal & SNMP**
2. Enable **SSH service**
3. Set **Port**: 22 (or custom port)
4. Click **Apply**

**Step 2: SSH into Synology**

```bash
ssh admin@synology.local
# Enter admin password
```

**Step 3: Create Directory and Run Setup**

```bash
# Create directory on volume
sudo mkdir -p /volume1/darkpipe
cd /volume1/darkpipe

# Download setup tool
curl -LO https://github.com/trek-e/darkpipe/releases/latest/download/darkpipe-setup-linux-amd64
chmod +x darkpipe-setup-linux-amd64

# Run setup
./darkpipe-setup-linux-amd64 setup
```

**Step 4: Deploy via Docker Compose**

```bash
# Start services
sudo docker compose up -d

# Check status
sudo docker compose ps

# View logs
sudo docker compose logs -f
```

## Platform-Specific Notes

### Container Manager vs Docker Package

**Container Manager (DSM 7.2+):**
- Native Docker Compose support
- Project-based UI
- Easier to manage
- **Recommended**

**Docker Package (DSM 7.0-7.1, deprecated):**
- Basic Docker support
- No Compose UI
- CLI-only for Compose
- Upgrade to DSM 7.2+ if possible

### Storage: Volume vs SHR

**Synology Volume:**
- Standard filesystem (ext4, btrfs)
- Works with all Docker volumes

**Synology Hybrid RAID (SHR):**
- RAID-like redundancy
- Recommended for production mail server
- Protects against drive failure

DarkPipe works with both. SHR recommended for data protection.

### Docker Secrets on Synology

Synology Container Manager supports Docker secrets via file-based secrets:

```yaml
secrets:
  admin_password:
    file: ./secrets/admin_password.txt
```

The setup tool creates secrets with correct permissions.

**Note**: Secrets may not show in Container Manager UI, but work correctly via Compose.

### Port Mapping

Synology Container Manager maps ports automatically from `docker-compose.yml`:

```yaml
ports:
  - "25:25"   # SMTP
  - "587:587" # Submission
  - "993:993" # IMAPS
  - "8080:8080" # Webmail
```

**Conflict resolution:**
If port is already in use (e.g., port 25 by Synology Mail Server):
1. Stop conflicting service in **Package Center**
2. Or: Change port mapping in docker-compose.yml (e.g., `2525:25`)

### Memory Limits

Synology Container Manager respects Docker Compose memory limits:

```yaml
deploy:
  resources:
    limits:
      memory: 512M
```

Monitor memory via **Resource Monitor** package:
1. Install **Resource Monitor** from Package Center
2. Check **Docker** tab for per-container memory usage

### Well-Known URLs for CalDAV/CardDAV

Synology may serve its own well-known URLs if Calendar/Contacts packages are installed.

To fix conflicts:
1. Uninstall Synology Calendar/Contacts packages
2. Or: Configure Caddy to handle well-known URLs (DarkPipe setup handles this)

### Backups

**Synology Hyper Backup:**
1. Install **Hyper Backup** package
2. Create backup task
3. Select `darkpipe` shared folder
4. Choose destination (external drive, cloud, another Synology)
5. Schedule regular backups

**Snapshot Replication (btrfs only):**
1. Install **Snapshot Replication** package
2. Create snapshot schedule for `darkpipe` folder
3. Instant recovery from snapshots

## Troubleshooting

### Container Manager shows "Failed to create project"

**Symptom**: Error when creating project with docker-compose.yml

**Cause**: Invalid YAML syntax or unsupported Compose features

**Fix**:
1. Validate docker-compose.yml syntax:
   ```bash
   docker compose config
   ```
2. Check Container Manager logs in **Log Center**
3. Try creating via CLI instead of UI

### Port 25 already in use

**Symptom**: Container fails to start with "port already in use"

**Cause**: Synology Mail Server or MailPlus using port 25

**Fix**:
1. Navigate to **Package Center**
2. Stop **Mail Server** or **MailPlus** package
3. Or: Change DarkPipe to use alternate port (not recommended for mail)

### Services show "Unhealthy" in Container Manager

**Symptom**: Containers restart frequently, logs show health check failures

**Cause**: Services haven't finished starting, or insufficient resources

**Fix**:
1. Wait 2-3 minutes for services to initialize
2. Check logs for errors
3. Increase memory allocation if needed
4. Verify DNS records are configured

### Can't access webmail from network

**Symptom**: Can ping Synology, but can't access port 8080

**Cause**: Firewall blocking port

**Fix**:
1. Navigate to **Control Panel** → **Security** → **Firewall**
2. **Edit Rules** for LAN interface
3. **Add** rule: Allow port 8080 from LAN
4. Click **OK**

### Docker Compose version mismatch

**Symptom**: `docker compose` not found or version error

**Cause**: Older DSM using `docker-compose` v1 instead of `docker compose` v2

**Fix** (DSM 7.2+):
```bash
# Container Manager includes Compose v2
# Use via Container Manager UI, or:
sudo /var/packages/ContainerManager/target/usr/bin/docker-compose version
```

**Fix** (DSM 7.0-7.1):
Upgrade to DSM 7.2+ for native Compose v2 support.

## Performance Benchmarks

**Synology DS920+ (4-core Celeron, 8GB RAM, SSD cache):**
- Mail delivery: <100ms
- IMAP FETCH: 10-30ms per message
- Webmail page load: 200-400ms
- RAM usage (Stalwart + SnappyMail): ~1.2GB
- Idle CPU: <5%

**Synology DS220+ (2-core Celeron, 2GB RAM, HDD only):**
- Mail delivery: 300-600ms
- IMAP FETCH: 50-150ms per message
- Webmail page load: 1-3 seconds
- RAM usage (Maddy + no webmail): ~600MB
- Idle CPU: <10%

## Security Considerations

1. **Network isolation**: Use Synology Firewall to restrict access
2. **Backups**: Enable Hyper Backup for regular backups
3. **Snapshots**: Use Snapshot Replication (btrfs volumes) for instant recovery
4. **Updates**: Keep DSM and Container Manager updated
5. **Firewall**: Configure firewall to only allow required ports

## Alternative Runtimes

Synology DSM uses its own Container Manager package, which is built on Docker. Podman is **not applicable** on Synology — the platform does not support installing or running alternative container runtimes. Use Container Manager (Docker) as described in this guide.

For Podman deployment on supported platforms, see the [Podman Platform Guide](podman.md).
